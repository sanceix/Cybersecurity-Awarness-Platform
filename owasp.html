OWASP Top 10

The most critical web application security risks, explained simply

Top 10 Web Application Security Risks

Each year OWASP (the Open Web Application Security Project) publishes the top ten security vulnerabilities. It represents a broad consensus about the most critical security risks to web applications.

Click on any vulnerability below to learn how it works, see real-world examples, and understand how to protect against it.

A01

Broken Access Control

Critical

Access control enforces policy such that users cannot act outside their intended permissions. Failures typically lead to unauthorized information disclosure, modification, or destruction of data.

How It Works

Attackers exploit flaws in access control mechanisms to bypass authorization. This can include modifying URLs, API requests, or internal application state to access unauthorized resources. For example, changing /user/profile?id=123 to id=456 to view another user's data.

Real-World Example

In 2019, a major financial institution exposed over 100 million customer records because their web application failed to properly restrict access to an internal API endpoint. Any authenticated user could access any other user's data by simply changing a parameter.

Common Weaknesses

  • Bypassing access control checks by modifying URLs or parameters
  • Viewing or editing someone else's account by providing its identifier
  • Accessing API endpoints with missing access controls for POST, PUT, DELETE
  • Elevation of privilege — acting as a user without being logged in, or as admin when logged in as a regular user

How To Prevent

  • Deny access by default — implement server-side access control
  • Use Role-Based Access Control (RBAC) consistently
  • Disable web server directory listing and remove sensitive files from web roots
  • Log access control failures and alert administrators
  • Rate limit API and controller access to minimize automated attack damage
A02

Cryptographic Failures

High

Previously known as "Sensitive Data Exposure," this category focuses on failures related to cryptography — or lack thereof — which often leads to exposure of sensitive data like passwords, credit cards, and personal information.

How It Works

Attackers steal sensitive data when applications fail to properly encrypt it. This includes data transmitted in clear text (HTTP, FTP, SMTP), using weak or old cryptographic algorithms, or storing passwords without proper hashing. Even encrypted data can be exposed if encryption keys are poorly managed.

Real-World Example

A healthcare provider stored patient records using MD5 hashing without salt. When their database was breached, attackers easily reversed millions of password hashes using rainbow tables, exposing sensitive medical data and patient identities.

Common Weaknesses

  • Transmitting data in clear text (HTTP instead of HTTPS)
  • Using deprecated hash functions like MD5 or SHA-1
  • Using default or weak cryptographic keys
  • Not enforcing encryption through security headers (HSTS)

How To Prevent

  • Classify and identify all sensitive data, and apply controls accordingly
  • Encrypt all data in transit with TLS and enforce it using HSTS
  • Use strong, up-to-date algorithms and protocols (AES-256, bcrypt)
  • Store passwords using strong salted hashing functions (Argon2, bcrypt)
  • Disable caching for responses containing sensitive data
A03

Injection

Critical

Injection flaws occur when an attacker can send hostile data to an interpreter as part of a command or query. SQL injection, NoSQL injection, OS command injection, and LDAP injection are among the most common attack vectors.

How It Works

When user input is not properly validated, filtered, or sanitized, attackers can inject malicious commands into queries. For example, entering ' OR 1=1 -- into a login form could bypass authentication entirely if the application builds SQL queries using string concatenation.

Real-World Example

In the infamous SQL injection attack on a US voter registration database, attackers used automated tools to inject SQL commands through search fields, extracting personal data of over 200,000 voters including names, addresses, and partial SSNs.

Common Weaknesses

  • User-supplied data not validated, filtered, or sanitized
  • Dynamic queries built with string concatenation
  • Hostile data used within ORM search parameters
  • Hostile data directly concatenated into SQL commands

How To Prevent

  • Use parameterized queries and prepared statements
  • Use positive server-side input validation
  • Escape special characters for any residual dynamic queries
  • Use LIMIT and other SQL controls to prevent mass disclosure
  • Use ORMs — but be aware they can still be vulnerable
A04

Insecure Design

High

Insecure design represents weaknesses in the application's architecture and design itself. Unlike implementation bugs, these are flaws in the fundamental security logic — no amount of perfect coding can fix a flawed design.

How It Works

Insecure design occurs when developers don't consider security during the planning phase. For example, designing a password reset flow that relies only on security questions (easily guessable or found on social media) without additional verification steps.

Real-World Example

A cinema chain allowed unlimited seat bookings without requiring payment upfront. Scalpers wrote bots to bulk-reserve premium seats for popular movies, then released unclaimed ones at the last minute — causing massive revenue loss and customer frustration.

Common Weaknesses

  • No threat modeling during design phase
  • Missing rate limiting on critical business flows
  • Trusting client-side validation alone
  • Not considering abuse cases alongside use cases

How To Prevent

  • Establish a secure development lifecycle with security professionals
  • Use threat modeling for authentication, access control, and business logic
  • Write unit and integration tests for security-critical flows
  • Separate tier layers and enforce trust boundaries
  • Limit resource consumption by user or service
A05

Security Misconfiguration

High

Security misconfiguration is the most commonly seen issue. This is commonly a result of insecure default configurations, incomplete configurations, open cloud storage, misconfigured HTTP headers, or verbose error messages containing sensitive information.

How It Works

Attackers scan for unpatched systems, default accounts, unused pages, unprotected files, and misconfigured services. For example, a server running with default admin credentials (admin/admin) or a cloud storage bucket left publicly accessible.

Real-World Example

Multiple high-profile breaches have occurred through misconfigured AWS S3 buckets. Companies accidentally left sensitive data including customer records, internal documents, and credentials in publicly readable cloud storage — discoverable through simple automated scanning.

Common Weaknesses

  • Unnecessary features enabled (ports, services, pages, accounts)
  • Default accounts and passwords still active
  • Error handling reveals stack traces or sensitive info
  • Security settings in frameworks and libraries not set to secure values

How To Prevent

  • Implement a repeatable, automated hardening process
  • Remove or don't install unused features, frameworks, and components
  • Review and update configurations as part of patch management
  • Use segmented application architecture for separation between components
  • Send security directives to clients (Security Headers)
A06

Vulnerable & Outdated Components

Medium

Applications using components with known vulnerabilities may undermine defenses and enable various attacks. Components run with the same privileges as the application, so flaws in any component can be serious.

How It Works

If you don't know the versions of all components you use (both client-side and server-side), or if any components are outdated, unsupported, or have known vulnerabilities, attackers can exploit these using publicly available exploit code.

Real-World Example

The Log4Shell vulnerability (CVE-2021-44228) in Apache Log4j affected millions of applications worldwide. Attackers could achieve remote code execution simply by sending a crafted string to any application that logged user input using the vulnerable library.

Common Weaknesses

  • Not knowing versions of all components and nested dependencies
  • Using software that is unsupported or out of date
  • Not scanning for vulnerabilities regularly
  • Not fixing or upgrading underlying platforms and libraries in a timely fashion

How To Prevent

  • Remove unused dependencies and unnecessary features
  • Continuously inventory all component versions (e.g., using SBOM)
  • Monitor sources like CVE and NVD for vulnerabilities
  • Only obtain components from official sources over secure links
  • Use tools like Dependabot, Snyk, or npm audit for automated scanning
A07

Identification & Authentication Failures

Critical

Confirmation of a user's identity, authentication, and session management is critical. Authentication weaknesses can allow attackers to compromise passwords, keys, or session tokens, or to exploit implementation flaws to assume other users' identities.

How It Works

Attackers exploit weak authentication by using brute force, credential stuffing (testing stolen username/password pairs), or session hijacking. If an application allows weak passwords like "password123" or doesn't implement account lockout, automated tools can crack accounts in seconds.

Real-World Example

A major social media platform's API allowed unlimited login attempts without rate limiting. Attackers used credential stuffing with a database of 500 million leaked passwords, successfully compromising millions of accounts that reused passwords.

Common Weaknesses

  • Permitting brute force or automated credential stuffing attacks
  • Allowing default, weak, or well-known passwords
  • Using plain text or weakly hashed passwords
  • Missing or ineffective multi-factor authentication
  • Exposing session identifiers in the URL

How To Prevent

  • Implement multi-factor authentication to prevent automated attacks
  • Do not deploy with default credentials
  • Check passwords against the top 10,000 worst passwords list
  • Limit or increasingly delay failed login attempts
  • Use a server-side, secure session manager that generates random session IDs
A08

Software & Data Integrity Failures

High

Software and data integrity failures relate to code and infrastructure that does not protect against integrity violations. This includes using software from untrusted sources, insecure CI/CD pipelines, and auto-update mechanisms without verification.

How It Works

Attackers compromise the software supply chain — injecting malicious code into libraries, plugins, or update mechanisms. If an application auto-updates without verifying the integrity (digital signatures, checksums), a compromised update server can push malware to all users.

Real-World Example

The SolarWinds supply chain attack (2020) compromised the Orion software build process. Attackers injected a backdoor into legitimate updates that was then distributed to over 18,000 organizations, including US government agencies and Fortune 500 companies.

Common Weaknesses

  • Using libraries from untrusted CDNs or repositories without integrity checks
  • Insecure CI/CD pipelines allowing unauthorized code changes
  • Auto-update functionality without sufficient integrity verification
  • Unsigned or unencrypted serialized data sent to untrusted clients

How To Prevent

  • Use digital signatures to verify software and data are from the expected source
  • Ensure libraries are consumed from trusted repositories
  • Use software supply chain security tools like OWASP Dependency-Check
  • Ensure your CI/CD pipeline has proper segregation, configuration, and access control
  • Do not send unsigned or unencrypted serialized data to untrusted clients
A09

Security Logging & Monitoring Failures

Medium

Without logging and monitoring, breaches cannot be detected. Insufficient logging, detection, monitoring, and active response occurs when auditable events are not logged, warnings and errors generate no alerts, and logs are only stored locally.

How It Works

When security events (logins, failed logins, high-value transactions) aren't properly logged and monitored, attackers can operate undetected for extended periods. The average time to identify a breach is 287 days — logging failures extend this dramatically.

Real-World Example

A major retailer was breached through their HVAC vendor's credentials. The attackers were active in the network for weeks, exfiltrating 40 million credit card numbers. The breach was eventually detected not by the company's security team, but by the US Department of Justice.

Common Weaknesses

  • Auditable events (logins, failed logins, transactions) not logged
  • Warnings and errors generate no or unclear log messages
  • Logs are only stored locally without centralized monitoring
  • No alerting thresholds or response escalation processes in place

How To Prevent

  • Ensure all login, access control, and server-side validation failures are logged
  • Ensure logs are generated in a format suitable for SIEM tools
  • Establish effective monitoring and alerting with low response times
  • Establish or adopt an incident response and recovery plan
  • Use tamper-evident logging mechanisms (append-only databases)
A10

Server-Side Request Forgery (SSRF)

Critical

SSRF flaws occur when a web application fetches a remote resource without validating the user-supplied URL. This allows an attacker to coerce the application to send a crafted request to an unexpected destination, even when protected by a firewall or VPN.

How It Works

Attackers supply or modify a URL that the server-side code reads or submits. This can trick the server into accessing internal services (like http://169.254.169.254/ for cloud metadata), scanning internal ports, or accessing data behind the firewall that should not be externally accessible.

Real-World Example

The 2019 Capital One breach exploited an SSRF vulnerability in a misconfigured web application firewall. The attacker sent crafted requests that made the server access AWS metadata service, obtaining IAM credentials that were used to access S3 buckets containing 100 million customer records.

Common Weaknesses

  • Fetching remote resources based on user-supplied URLs
  • Not validating or sanitizing URLs before server-side fetching
  • Allowing access to internal services (localhost, metadata APIs)
  • Not blocking outbound traffic to unexpected destinations

How To Prevent

  • Sanitize and validate all client-supplied input data (URL allowlists)
  • Enforce "deny by default" firewall policies for network traffic
  • Disable HTTP redirections and use an allowlist for DNS resolution
  • Do not send raw responses to clients — validate response type
  • Segment remote resource access into separate networks
